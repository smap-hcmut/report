#### 4.7.6.1 Tổng quan

Ở cấp độ Container (4.7.5), chúng ta đã thấy các services như "hộp đen". Ở cấp độ Component (Level 3), chúng ta "mở hộp" này ra để xem bên trong mỗi service phức tạp được tổ chức như thế nào.

**Services cần phân tích chi tiết:**
- Analytics Service (Python) - Service phức tạp nhất về AI/ML
- Collection Service (Golang Manager + Python Workers) - Service phức tạp nhất về orchestration

**Services khác (không cần diagram chi tiết):**
- Identity, Project, Insight Services: Đơn giản hơn, theo pattern CRUD chuẩn
- Đã mô tả đầy đủ trong 4.7.5.3

**Component Responsibilities:**

| Component | Trách nhiệm | Input | Output | Technology |
|-----------|-------------|-------|--------|------------|
| **Event Consumer** | Nhận events từ RabbitMQ | `data.collected` event | Job metadata | pika (AMQP client) |
| **Orchestrator** | Điều phối pipeline | Job metadata | Workflow state | Python async |
| **Data Loader** | Tải dữ liệu từ MinIO | Object key | Raw posts (stream) | boto3 (S3 client) |
| **Preprocessing** | Làm sạch dữ liệu | Raw text | Clean tokens | NLTK, underthesea |
| **Sentiment Analyzer** | Phân tích cảm xúc | Text tokens | Sentiment scores | PyTorch + PhoBERT |
| **NER Component** | Trích xuất entities | Text tokens | Entity list | Transformers + PhoBERT-NER |
| **Topic Modeling** | Phân loại chủ đề | Document corpus | Topic distribution | Gensim (LDA) |
| **Result Aggregator** | Gộp kết quả | Multiple results | Structured output | Pandas |
| **Result Publisher** | Lưu & publish | Structured output | Event + DB record | SQLAlchemy + pika |
| **Health Monitor** | Giám sát sức khỏe | Service metrics | Prometheus metrics | prometheus_client |

**Data Flow trong Analytics Service:**

```
1. Event Consumer nhận event "data.collected"
   ↓
2. Orchestrator tạo processing job
   ↓
3. Data Loader stream data từ MinIO
   ↓
4. Preprocessing làm sạch text
   ↓
5. Parallel Processing:
   ├─▶ Sentiment Analyzer: PhoBERT → Pos/Neg/Neutral
   ├─▶ NER Component: Extract brands, KOCs, locations
   └─▶ Topic Modeling: LDA → 20 topics
   ↓
6. Result Aggregator merge tất cả kết quả
   ↓
7. Result Publisher:
   ├─▶ Save to PostgreSQL (sentiment_results table)
   ├─▶ Publish "analysis.done" event
   └─▶ Update Redis (job status: completed)
```

**Component Responsibilities:**

| Component | Trách nhiệm | Technology | Notes |
|-----------|-------------|------------|-------|
| **Event Listener** | Nhận "project.created" từ Project Service | pika (Go AMQP) | Auto-trigger crawl |
| **Job API** | REST endpoints cho CRUD jobs | Gin framework | Manual job creation |
| **Job Scheduler** | Phân tách job thành tasks, priority queue | Redis (sorted set) | Cron-based + event-driven |
| **Task Dispatcher** | Phân phối tasks cho workers qua RabbitMQ | AMQP | Load balancing |
| **Job Repository** | Lưu trữ metadata jobs | MongoDB | Job lifecycle tracking |
| **Redis Lock** | Distributed lock cho job processing | go-redis | Prevent duplicate processing |
| **Rate Limiter** | Token bucket cho mỗi platform | Redis (INCR) | Respect API limits |

**Worker Components:**

| Component | TikTok Worker (Python) | YouTube Worker (Go) | Facebook Worker (Go) |
|-----------|----------------------|-------------------|-------------------|
| **Consumer** | Celery worker | Go goroutine | Go goroutine |
| **Crawler** | Playwright (browser automation) | HTTP client (API calls) | Graph API client |
| **Anti-bot** | ✅ (stealth mode, signatures) | ❌ (official API) | ❌ (official API) |
| **Proxy** | ✅ (residential proxies) | ⚠️ (optional) | ⚠️ (optional) |
| **Transform** | JSON normalization | JSON normalization | JSON normalization |
| **Throughput** | 10k posts/min | 6.25k posts/min | 16.6k posts/min |

**Workflow:**

```
1. Project Service publishes "project.created" event
   ↓
2. Collection Manager's Event Listener receives event
   ↓
3. Job Scheduler creates CrawlJob:
   - Split into tasks per platform
   - Assign priority (urgent campaigns first)
   - Save to MongoDB
   ↓
4. Task Dispatcher publishes tasks to RabbitMQ:
   - tiktok.tasks queue
   - youtube.tasks queue
   - facebook.tasks queue
   ↓
5. Workers consume tasks (10 pods/platform):
   - Check rate limit (Redis)
   - Execute crawl
   - Transform data
   - Bulk insert to MongoDB (1000 docs/batch)
   ↓
6. After batch completion:
   - Export to MinIO (50k posts → 100MB file)
   - Publish "data.collected" event
   - Update job status in MongoDB
```

**Scaling Strategy:**

| Scenario | Action | Result |
|----------|--------|--------|
| Viral campaign (10x traffic) | HPA scales workers 2→20 pods | Handle 450k posts/hour |
| API rate limit hit | Exponential backoff | Graceful degradation |
| Worker crash | RabbitMQ re-queues unacked tasks | Zero data loss |
| MongoDB overload | Connection pooling (10 connections) | Smooth write throughput |

#### 4.7.6.4 Design Patterns Applied

**Analytics Service:**

| Pattern | Where Applied | Benefit |
|---------|---------------|---------|
| **Pipeline Pattern** | Data flow through stages | Clear separation of concerns |
| **Strategy Pattern** | Multiple ML models (Sentiment, NER, Topic) | Easy to add new models |
| **Observer Pattern** | Event consumer → processing | Loose coupling with Collection |
| **Repository Pattern** | Result persistence | Abstract database operations |

**Collection Service:**

| Pattern | Where Applied | Benefit |
|---------|---------------|---------|
| **Master-Worker Pattern** | Manager + multiple workers | Horizontal scalability |
| **Command Pattern** | Task dispatching | Encapsulate crawl requests |
| **Factory Pattern** | Create platform-specific workers | Polymorphic worker creation |
| **Circuit Breaker** | API call failures | Prevent cascading failures |

#### 4.7.6.5 Deployment Considerations

**Analytics Service:**
- **Replicas:** 2-5 pods (HPA based on queue depth)
- **Resources:** 1 CPU, 4GB RAM (CPU mode) or 1 GPU + 8GB RAM (GPU mode)
- **Health checks:** `/health` endpoint, model loading verification

**Collection Service:**
- **Manager:** 2 pods (lightweight, stateless)
- **Workers:** 1-10 pods per platform (HPA based on RabbitMQ queue depth)
- **Resources:** 250m CPU, 512MB RAM per worker

#### 4.7.6.6 Summary

**Key Takeaways:**

✅ **Analytics Service:**
- Clean separation: Presentation → Application → Domain → Infrastructure
- ML models isolated in Domain layer
- Easy to test and replace models

✅ **Collection Service:**
- Master-Worker pattern scales horizontally
- Platform-specific workers (polyglot: Python + Go)
- Fault-tolerant with RabbitMQ message durability
